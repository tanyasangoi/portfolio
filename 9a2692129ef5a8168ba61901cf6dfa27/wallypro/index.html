Wally Pro


Role:

UX Design Intern

Tools:

Adobe XD

Axure

Usability Testing

Duration:

2 months

Wally Pro was an enterprise dashboard that allowed a user to monitor and manage alerts related to thousands of sensors. This involved a lot of information overload for the user of the application. My task for this project was to simplify the dashboard and help the user manage multiple alerts smoothly.



Problem Background Process Solution Outcome

Problem

The current Wally Dashboard presented an overwhelming amount of information to a user. Currently the users struggle to manage multiple alerts that arise simultaneously due to the large number of sensors. There is also a need to simplify the drill-down from the dashboard view to the sensor view where the alerts are resolved. In addition, information must be presented in a way that it is easy to consume and allows for the user to take any necessary actions.

Background

The Wally Pro application is an enterprise application that monitors multiple units and locations through one central dashboard. It helped monitor multiple sensors which tracked a variety of metrics such as water leaks, humidity, and temperature.

The Wally Pro application was owned by the building management while the Wally devices installed in apartments could be owned by either the resident or the management. The residents could monitor their local devices using a mobile application while the building staff used the dashboard to get a global overview of the system.

With the increasing size of the managed properties, there was an increase in number of monitored sensors which added to the complexity of the system. For example, In a large residential complex Wally Pro could be used by the building staff to monitor multiple apartments simultaneously. The building staff could use the dashboard to track any alerts triggered by the sensors and take action accordingly.

The Wally Pro enabled a drill-down navigation view of the entire system. The dashboard view provided a summary of the recent events. The unit level view provided an overview of events related to each unit. Finally, the sensor level view which was the most granular view in that it provided events specific to each sensor.

Process

Ethnographic Research

My colleagues conducted ethnographic research to learn more about the Wally customer and how the customer used Wally in their environment. They interviewed 4 onsite employees at a property in Los Angeles: The General Manager, Building Engineer, and Front Desk (Supervisor & Lead Concierge) and learnt about the challenges that each of them faced. They observed participants as they conducted their daily activities and asked questions when appropriate regarding their daily life tasks, about their role and use of Wally.

Insights

These were some of the insights derived from the Ethnographic Study that pertained to the Wally Pro application

Dashboard Page
  1. Overview: The current system is unable to provide an actionable view of the system events. Users want to be able to filters by type of event, property, unit, floors and date.

  2. Managing Properties/Units: The building management want the ability to add and edit a property or apartment unit. When residents move out with their Wally devices, they want to able to delete those devices from the system. In the case that the devices were owned by the building management, they want to able to transfer the devices to the next resident.

  3. Managing Alerts: The users should be able to immediately see which units have an active event. Currently, this critical information is hidden and not easy to digest when multiple events are displayed.

Properties / Units Page
  1. Ineffective Presentation: The users complained that the cards were ineffective at communicating the alerts pertaining to each unit. When the users were unable to search for a unit, they had to scroll quite a bit. Instead of provided the average sensor statistics, they wished for the exact statistics of each sensor. It was also hard to get an overview of multiple units and prioritize which alerts should be resolved. A lot of the users also said that the label for this page should be changed to “units” as it was more consistent with how they spoke.

Open Events Page
  1. Inability to Track Status: Open Events are events that are not resolved. These events are resolved and disappear from the list when the building maintenance takes corrective action. Currently, the building management is unable to track the progress of repair. It would be extremely useful if they could view the current status of the event so that they could take appropriate action and improve transparency.

Recent Events Page
  1. Confusing Labels: Recent events are events that have been recently resolved by the building staff. Currently, the dashboard users are confused between open and recent events. Elements in this page, the call to action button in particular, do not sufficiently indicate to the user that the event has been resolved. The users appreciate the ability to see event history but they want to see it in the form of reports in which can customized by unit or time. This could enable them to see trends and recurring issues.

Sensor Page
  1. Multiple Alert Resolution: The sensor page is where the alerts are resolved. It is very hard to comprehend what the current status of resolution is. The users want the sensor page to have actionable insights. When a new alert occurs, there is a popup that asks the user to either investigate now or to take action later. If the user chooses to investigate the alert now they will lose the progress of the alert that they were in the process of resolving. Thus they want a capability to store their current progress and ease of navigation back to alert they were working on before.

Sketching

I made notes of the various revelations from the ethnographic study and compiled a list of questions for engineering to answer. I then began sketching a few ideas that help the main objectives of the user. After every iteration I would brainstorm with my manager and try to discover ways to improve the current design. Once we agreed upon the design, the final sketch was converted to high fidelity prototype.

Solution

Dashboard Page

The Dashboard page was modified to provide an overview of multiple events simultaneously. The status of resolution helped expedite the decision making process by the building staff. By clicking on the alerts page the user would be directed straight to the sensor page where alerts get resolved.

Units Page

The Units page provided the analytics of the recent events in the system. The buildings which were in the process of onboarding Wally Pro had requested a way to track the progress of installation of these devices in individual units. We used radial gauge to present the installation progress. The management was also able to see the amount of money they had saved through the timely detection of various events. It also provided an overview of all the sensors in the unit and their status.

The new redesign also provided the user the affordance to control account activation for individual units unlike the previous iteration.

Sensor Page

The sensor page was updated to be able to provide more actionable insights. The new redesign made it possible to assign status of the resolution. The notes section of the previous design was renamed to comments to capture a detailed explanation of what actions were taken. It also automatically included a timestamp of when this update was made. The user now had the capability to capture event resolution from the dashboard.

Events Page

The events page was synonymous with the recent events page. It captured all the events that had occured within a given time frame. The user could use various filters to narrow down their search.
